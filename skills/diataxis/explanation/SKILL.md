# Explanation 문서 작성 가이드

> 참고: [Diátaxis - Explanation](https://diataxis.fr/explanation/)

## 정의

Explanation은 **배경 지식과 이해를 제공**합니다.
- 목적: "왜?"에 대한 답 — 깊은 이해
- 비유: 요리의 과학 vs 레시피 — 원리를 이해하면 응용 가능

## 다른 유형과의 차이

| 유형 | 질문 | 관점 |
|------|------|------|
| Tutorial | "어떻게 배우지?" | 학습자 눈높이 |
| How-to | "어떻게 하지?" | 작업자 눈높이 |
| Reference | "이게 뭐지?" | 코드 눈높이 |
| **Explanation** | "**왜 이렇지?**" | **높은 관점, 넓은 시야** |

## 핵심 원칙

### DO
- **맥락 제공** — 역사, 배경, 대안
- **연결** — 다른 개념/시스템과의 관계
- **설계 결정의 이유** 설명 — "왜 A 대신 B인가?"
- **트레이드오프** 논의 — 장단점, 제약조건
- **여러 관점** 제시 — 다양한 시각

### DON'T
- 단계별 지시 X → How-to로
- 상세 스펙 X → Reference로
- 따라하기 X → Tutorial로

## 구조 템플릿

### 아키텍처 설명
```markdown
# [시스템/개념] 이해하기

## 개요
[시스템이 해결하는 문제와 접근 방식 요약]

## 배경
[왜 이 시스템이 필요했는가? 역사적 맥락]

## 핵심 개념
### [개념 1]
[개념 설명과 시스템에서의 역할]

### [개념 2]
[개념 설명과 다른 개념과의 관계]

## 설계 결정
### 왜 [선택 A]인가?
고려한 대안:
- [대안 1]: [장단점]
- [대안 2]: [장단점]

선택 이유: [근거]

## 제약사항과 트레이드오프
- [제약 1]: [영향]
- [트레이드오프 1]: [얻은 것] vs [잃은 것]

## 관련 개념
- [외부 개념/시스템과의 연결]
```

### 의사결정 기록 (ADR)
```markdown
# ADR-[번호]: [제목]

## 상태
[Proposed | Accepted | Deprecated | Superseded by ADR-X]

## 맥락
[결정이 필요했던 상황과 배경]

## 결정
[무엇을 결정했는가]

## 근거
[왜 이 결정을 내렸는가]

### 고려한 대안
1. **[대안 1]**
   - 장점: ...
   - 단점: ...

2. **[대안 2]**
   - 장점: ...
   - 단점: ...

## 결과
[이 결정으로 인한 영향 — 긍정적, 부정적 모두]

## 관련 문서
- [관련 ADR]
- [관련 Explanation]
```

## 품질 체크리스트

- [ ] **"왜?"에 대한 답**이 있는가?
- [ ] **맥락과 배경**이 충분한가?
- [ ] **설계 결정의 근거**가 명확한가?
- [ ] **대안과 트레이드오프**가 논의되었는가?
- [ ] 다른 개념과의 **연결**이 있는가?
- [ ] 단계별 지시나 상세 스펙이 **없는가**?

## 언제 Explanation을 작성하는가?

- 새로운 아키텍처 도입 시
- 중요한 기술 선택 시
- 복잡한 시스템 설명이 필요할 때
- "왜 이렇게 했어요?" 질문이 반복될 때
