# Functionality Review Skill

## 목적
코드가 의도한 대로 작동하며, 사용자(개발자 포함)에게 좋은 경험을 제공하는지 평가합니다.

## 핵심 질문

1. **의도 일치**: 코드가 작성자의 의도대로 동작하는가?
2. **사용자 경험**: 최종 사용자에게 좋은가?
3. **개발자 경험**: 다른 개발자가 사용하기 쉬운가?

## 검토 기준

### 정확성
| 체크 | 기준 |
|------|------|
| [ ] | 요구사항을 충족함 |
| [ ] | 엣지 케이스 처리됨 |
| [ ] | 에러 상황 적절히 처리됨 |
| [ ] | 경계값 검증됨 |

### 안전성
| 체크 | 기준 |
|------|------|
| [ ] | Null/None 체크 |
| [ ] | 범위 초과 방지 |
| [ ] | 타입 안전성 |
| [ ] | 예외 전파 적절함 |

### 특별 주의 영역

#### UI 변경
- 사용자 흐름에 문제없는가?
- 접근성(a11y) 고려되었는가?
- 반응형 디자인 적용되었는가?

#### 병렬/비동기 처리
| 체크 | 기준 |
|------|------|
| [ ] | Race condition 없음 |
| [ ] | Deadlock 가능성 없음 |
| [ ] | 리소스 정리(cleanup) 보장 |
| [ ] | 취소(cancellation) 처리 |

## Red Flags (경고 신호)

- 🐛 테스트 없이 복잡한 조건문
- ⚡ 검증 없는 외부 입력 사용
- 🔒 잠금 없는 공유 상태 변경
- 💥 catch-all 예외 처리 (`except:` / `catch (e)`)
- 🔄 무한 루프 가능성

## 질문 예시

```markdown
**[Critical]** src/api/handler.py:123

`user_input`이 직접 SQL 쿼리에 사용됩니다. SQL Injection 취약점이 있습니다.
파라미터화된 쿼리를 사용해주세요:

```python
# Before
cursor.execute(f"SELECT * FROM users WHERE id = {user_input}")

# After
cursor.execute("SELECT * FROM users WHERE id = ?", (user_input,))
```
```

## DO / DON'T

### DO
- 실제 실행 시나리오 상상하며 검토
- 엣지 케이스 구체적으로 지적
- 보안 취약점은 Critical로 표시

### DON'T
- 모든 경로를 직접 테스트하려 하지 않음 (테스트 코드 검토로 대체)
- 동작하는 코드를 "더 나은 방식"으로 바꾸라 요구
