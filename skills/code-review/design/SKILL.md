# Design Review Skill

## 목적
코드의 전체적인 설계와 구조가 시스템에 적합한지 평가합니다.

## 핵심 질문

1. **소속**: 이 코드가 이 위치에 있어야 하는가?
2. **통합**: 기존 시스템과 잘 통합되는가?
3. **적시성**: 지금 이 변경이 필요한가?
4. **범위**: 변경 범위가 적절한가? (너무 크거나 작지 않은가)

## 검토 기준

### 구조적 적합성
| 체크 | 기준 |
|------|------|
| [ ] | 새 코드가 적절한 모듈/패키지에 위치함 |
| [ ] | 기존 아키텍처 패턴을 따름 |
| [ ] | 계층 구조를 위반하지 않음 (예: UI → Service → Repository) |
| [ ] | 순환 의존성이 없음 |

### 책임 분리
| 체크 | 기준 |
|------|------|
| [ ] | 단일 책임 원칙 준수 (한 클래스/함수는 한 가지 일) |
| [ ] | 관련 기능이 적절히 그룹화됨 |
| [ ] | 불필요한 커플링 없음 |

### 확장성
| 체크 | 기준 |
|------|------|
| [ ] | 향후 변경에 유연함 (단, 과도한 추상화 주의) |
| [ ] | 기존 코드 수정 없이 확장 가능 (Open-Closed) |

## Red Flags (경고 신호)

- 📁 한 파일에 여러 관련 없는 기능
- 🔄 양방향 의존성
- 🏗️ 거대한 클래스 (God Object)
- 🔗 과도한 상속 계층 (3단계 이상)
- 📦 잘못된 패키지/모듈 위치

## 질문 예시

```markdown
**[Minor]** src/utils/helpers.py:45

이 함수는 인증 관련 로직인데 `utils`보다 `auth` 모듈에 더 적합해 보입니다.
`src/auth/utils.py`로 이동하는 것은 어떨까요?
```

## DO / DON'T

### DO
- 기존 코드베이스의 패턴 파악 후 평가
- 구체적인 대안 위치 제시
- 현재 필요에 집중

### DON'T
- 미래의 가상 요구사항을 위한 설계 요구
- 개인 선호 스타일 강요
- 작동하는 코드를 이상적 구조로 재작성 요구
